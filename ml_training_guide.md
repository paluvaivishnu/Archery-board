# YOLO Model Training & Optimization Guide for Target Impact Recognition

This guide outlines a production-grade workflow to train and optimize your object detection model (specifically **YOLOv8** or **YOLO11**) using the synthetic dataset generated by the Bullseye AI platform.

---

## 📂 1. Dataset Structure Overview
When you export a dataset from the **ML Dataset Generator** tab, you obtain a ZIP archive containing:
```
dataset/
├── data.yaml            # Dataset configuration (paths and class names)
├── images/
│   └── train/           # Target board images (.png)
└── labels/
    └── train/           # Normalized YOLO coordinates (.txt)
```
Each annotation text file contains entries in the format:
`<class_id> <x_center> <y_center> <width> <height>` (all values normalized `0.0` to `1.0`).

## ⚡ 2. High-Efficiency Training Pipeline

### Step A: Setup Environment
Install the Ultralytics framework and training dependencies:
```bash
pip install ultralytics albumentations opencv-python-headless
```

### Step B: The Training Script
Create a Python script `train.py` to start training:
```python
from ultralytics import YOLO

def train_model():
    # 1. Load a pre-trained Nano model (highly efficient, ideal for real-time edge runtimes)
    model = YOLO("yolov8n.pt")  # Or yolov11n.pt

    # 2. Train with optimized hyperparameters
    model.train(
        data="datasets/fulltargethole.yolov11/data.yaml",    # Path to config yaml
        epochs=100,                  # Run for 100 epochs
        imgsz=640,                   # Train size (balances precision and speed)
        batch=16,                    # Adjust based on GPU memory (8, 16, 32)
        device="0",                  # Set to '0' for CUDA GPU, 'mps' for Apple Silicon, or 'cpu'
        workers=8,                   # Number of CPU loader threads
        cache=True,                  # Cache dataset in RAM for faster epoch transitions
        close_mosaic=10,             # Disable mosaic augmentation in final 10 epochs for accuracy
        save=True,                   # Save checkpoints and best models
        project="bullseye_ai",       # Project output directory
        name="impact_detector_yolo"   # Run name
    )

if __name__ == "__main__":
    train_model()
```

## 🛠️ 3. Advanced Optimization Techniques for Peak Efficiency

### 1. Hardware-Specific Acceleration
* **Apple Silicon (Mac M1/M2/M3)**: Set `device="mps"` to leverage the Apple Metal Performance Shaders (MPS) neural engine.
* **NVIDIA GPUs**: Use CUDA (`device="0"`) and verify mixed-precision FP16 training is enabled by default to reduce VRAM footprint by 50% and double execution speed.

### 2. Multi-Stage Transfer Learning (Fine-Tuning)
Instead of training from scratch:
1. Initialize training with standard weights pre-trained on the COCO dataset (`yolov8n.pt`).
2. The initial layer features (edges, corners, circles) are reused, drastically reducing the number of epochs required to reach high accuracy (typically converges within **30–50 epochs** instead of **300+**).

### 3. Smart Dataset Caching
Setting `cache=True` in the training arguments pre-loads images directly into your system memory (RAM). This eliminates storage I/O bottlenecks during epoch transitions, speeding up training times by **up to 3x** on standard HDD/SSD setups.

### 4. Small Object (Hole) Localization Optimizations
Bullet holes, air pellet holes, and arrow entries represent extremely small features relative to the overall target sheet. Standard YOLO training configurations are optimized for medium-to-large objects and may fail to identify these tiny structures. Apply these hyperparameters:
* **Train at High Resolution (`imgsz=1024` or `1280`):** Standard models train at `640`. Training at `1024` gives the network **2.5x more pixels** per hole, allowing it to capture micro-level edge details.
* **Turn Off Mosaic Augmentation Late (`close_mosaic=20`):** Mosaic augmentation blends 4 training images together, reducing the size of already-small holes even further. Disabling it for the final 20 epochs lets the model converge on clean, realistic target borders.
* **Tune Loss Gains (`box=2.0`, `cls=0.5`):** Increase the regression loss weight (`box`) to force precise coordination alignment, and reduce class classification weight (`cls`) to prevent classification gradients from dominating.

### 5. Model Sizing for Web Execution (Nano vs. Medium)
For web application execution via ONNX Runtime Web:
* **YOLO11 Nano (`yolo11n.pt`) [Recommended]:** Generates an **11MB** ONNX file. It runs in **100–200ms** on standard browser CPU/WASM environments, providing a smooth real-time user experience.
* **YOLO11 Medium (`yolo11m.pt`):** Generates an **80MB** ONNX file. Loading this file takes significant network bandwidth, consumes excessive RAM, and results in **1.5–2.5s** of CPU execution lag per image.
* **Decision:** Always train and deploy the **Nano (n)** or **Small (s)** models for edge devices and browser clients. Keep the Medium (m) models strictly for high-end server-side API runtimes.

## 🧪 4. Model Conversion for Web Execution
To run your model 100% client-side in the browser via JavaScript (similar to OpenCV.js), export the trained `.pt` model to **ONNX** format:

```python
from ultralytics import YOLO

# Load your best trained model
model = YOLO("bullseye_ai/impact_detector_yolo/weights/best.pt")

# Export to ONNX with half-precision quantization (FP16)
model.export(format="onnx", half=True, imgsz=640)
```
This produces a lightweight `best.onnx` file that can be loaded in web applications using `onnxruntime-web` for ultra-fast, offline inference.
